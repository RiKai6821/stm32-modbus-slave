# STM32 MODBUS RTU 从机协议栈

> 基于 STM32F103 的 MODBUS RTU 从机协议栈实现，从零开始用 C 语言实现，不依赖第三方 MODBUS 库。

## 项目简介

本项目实现了一个完整的 MODBUS RTU 从机协议栈，运行在 STM32F103 微控制器上。MODBUS RTU 是工业自动化领域应用最广泛的串行通信协议之一，广泛用于 PLC、传感器、变频器、HMI 等设备之间的通信。

本项目的目标是深入理解工业通信协议的设计思想，包括帧同步、CRC 校验、寄存器映射、异常处理等核心概念。

## 功能特性

- ✅ 完整实现 MODBUS RTU 从机协议
- ✅ 支持功能码 0x03（读保持寄存器）、0x06（写单个寄存器）、0x10（写多个寄存器）
- ✅ 标准 MODBUS CRC16 校验（查表法实现，效率高）
- ✅ 基于定时器的 3.5 字符帧间隔检测
- ✅ UART 中断接收 + 状态机式帧解析，主循环非阻塞
- ✅ 完整的异常响应机制（非法功能码、非法地址、非法数据）
- ✅ 运行统计（帧计数、CRC 错误数、异常数）
- ✅ 提供完整的 Python 测试脚本

## 帧处理流程

1. **接收**：UART 收到一字节 → 中断回调 → 重置 3.5 字符定时器
2. **断帧**：定时器超时 → 标记帧完成 → 状态切换为 `FRAME_READY`
3. **校验**：主循环检查状态 → 验证 CRC16 → 不通过则丢弃
4. **分发**：根据功能码分发到对应处理函数
5. **响应**：组装响应帧 → 通过 UART 发送

## 寄存器映射

| 地址范围 | 用途 | 访问 |
|---------|------|------|
| 0x0000 - 0x000F | 系统状态（版本、运行计数、状态字） | 只读 |
| 0x0010 - 0x001F | 控制寄存器（命令、参数） | 读写 |
| 0x0020 - 0x002F | 模拟数据（温度、电压等） | 只读（设备更新） |

## 编译与烧录

### 使用 STM32CubeIDE

1. 在 CubeMX 中配置：
   - USART1: 115200 8N1，中断使能
   - TIM2: 配置为 3.5 字符超时（115200bps 时约 305μs，分频系数自行计算）
   - PC13: 输出
2. 生成代码，将 `modbus_slave.c/h` 加入工程
3. 在 `main.c` 中按照本项目示例集成
4. 编译并通过 ST-Link 烧录

### 关于 3.5 字符超时定时器的计算

MODBUS RTU 规定帧间隔必须 >= 3.5 字符时间：

- 一个字符 = 1 起始 + 8 数据 + 1 停止 = 10 位
- 在 115200 bps 下，1 字符 = 86.8 μs，3.5 字符 ≈ 304 μs
- TIM2 时钟假设为 72MHz，分频到 1MHz（PSC = 71），ARR = 304 即可

## 技术细节

### 为什么不用 HAL 的 IDLE 中断？

STM32F103 的 USART IDLE 中断对于普通 MODBUS 应用够用，但其超时时间不可控（固定为 1 帧字符时间）。**MODBUS RTU 标准明确要求 3.5 字符帧间隔**，所以本项目使用独立定时器，超时时间精确可配。

### 为什么用查表法实现 CRC16？

逐位计算 CRC16 对每个字节需要 8 次循环，64 字节帧 = 512 次运算。查表法只需 64 次查表 + XOR，**性能提升约 8 倍**，对于 115200 bps 的高速 MODBUS 场景至关重要。

### 为什么用状态机而不是阻塞接收？

阻塞接收会让 CPU 一直卡在 `HAL_UART_Receive` 上，无法做其他工作。**状态机 + 中断 + 定时器**让协议栈在后台运行，主循环可以同时处理传感器采集、显示更新等任务，这是工业代码的标准写法。
